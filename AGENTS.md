## コーディング規約

### 言語・スタック
- JavaScript（ES2020+、モジュール化せず単一 `app.js` で管理）
- HTML/CSS は vanilla、**ビルドステップなし**
- Apps Script（`apps_script/Code.gs`）は V8 ランタイム想定

### スタイル
- インデント: **4スペース** 統一（app.js / style.css / index.html すべて）
- 文字列: 基本ダブルクォート、テンプレートリテラル使用可
- コメント: 日本語 OK。**WHY を残す**（WHAT はコード読めば分かる）
- 関数は短く保ち、セクション区切りに `// ==== セクション名 ====` を使う

### UI テキスト
- ユーザー向け文言は **日本語**
- AI プロンプト（Gemini 送信内容）は **英語** で書く
- トースト・エラーメッセージは **日本語**、簡潔に

### コード修正ポリシー
- 必要最小限のスコープで修正する。関係ない部分のリファクタは混ぜない
- コメントを削除しない（特に WHY コメント）
- 既存の命名慣習に従う

## コミット規約

Conventional Commits 風：

| Prefix | 意味 |
|---|---|
| `feat:` | 機能追加 |
| `fix:` | バグ修正 |
| `docs:` | ドキュメントのみ |
| `refactor:` | 挙動を変えないリファクタ |
| `perf:` | パフォーマンス改善 |
| `revert:` | 戻し |
| `chore:` | 雑多（設定、依存関係など） |

- 本文は日本語 OK、**WHY を重視**
- 1コミット1関心事（混ぜない）
- 破壊的変更や再デプロイ要件があれば本文に明記

## ブランチ運用

- **`main` のみが本番**。GitHub Pages は main から自動配信
- 通常の作業は main 直push（個人開発のため軽量運用）
- 大きな実験は `version/*` ブランチに保存（参考用スナップショット）
- 不要になったブランチは削除する

現存ブランチ:
- `main` — 本番
- `version/v1-bundled` — 4スタイル一括取得版の記念スナップショット（現行は段階的ロード）

## GAS 再デプロイが必要なタイミング

`apps_script/Code.gs` を変更した場合、**必ずユーザーに再デプロイを案内する**：

> 1. スプレッドシートの「拡張機能 > Apps Script」を開く
> 2. Code.gs を最新版に全置換して保存
> 3. 「デプロイ > デプロイを管理 > 鉛筆アイコン > バージョン: 新しいバージョン > デプロイ」
> 4. URL は変わらないのでアプリ側の設定変更は不要

フロントのみの変更（app.js / index.html / style.css）は GitHub Actions が自動デプロイするので案内不要。

## 既知の重要な制約・落とし穴

これらは **試行錯誤の末に判明した事項**。安易に触り直さない。

### iOS Safari の音声セッション問題
- `SpeechRecognition` を使うと iOS audio session が `PlayAndRecord` カテゴリに切り替わり、**その後の TTS が earpiece（通話スピーカー）から出る**ようになる
- JS から復旧する方法は **ないと結論済み**（AudioContext 無音バッファ / 無音ループ / オシレータ等すべて試した）
- 対処: 触らない。ユーザにはページリロードで回復する旨を伝える
- 詳細: DEVLOG の 2025-XX-XX エントリ参照

### `speechSynthesis.pause()` は禁止
- iOS で paused 状態に固まり、その後 `speak()` を呼んでも音が出なくなる
- 代わりに `killTTS()` は `cancel()` + 80ms 後に再 `cancel()` のみ使用
- `speakText()` / `speakResult()` 先頭で `resume()` セーフティ呼出

### ペアキー正規化
- `ja-en` と `en-ja` は同一ペアとして扱う
- pairKey は **アルファベット順**（例: `en-ja`）、langA/textA は先、langB/textB は後
- `makePairKey()` と `normalizePair()` を参照

### Inbox の既読管理
- `seen` 状態は **localStorage のみ**（DB に書かない、端末ローカル）
- 端末間同期はされない設計。処理すれば inbox 自体から消えるので実害なし

## 典型的な作業フロー（エージェント向け）

### 機能追加
1. ユーザー要件を確認し、論点があれば先に議論
2. 必要なら `TodoWrite` でタスク分解
3. コード修正 → テスト方法をユーザーに提示
4. `git commit` + `git push origin main`
5. GAS 変更があれば再デプロイ案内
6. 区切りで `/checkpoint`（DEVLOG.md に追記）

### バグ修正
1. 再現条件をユーザと確認
2. 原因の仮説を立てて共有（決めつけずに検証）
3. 最小修正 → コミット → プッシュ

### ドキュメント整備
- `docs:` prefix でコミット
- ユーザの要望をそのまま書くのではなく、読み手（未来の自分 or 別エージェント）が理解できる形に整形

## エージェント切替

作業エージェントを切り替える時は：

1. 現エージェントで作業の区切りをつけ `/checkpoint`
2. 新エージェント起動時、Codex なら「まず AGENTS.md を読んで」と最初に伝える
3. [DEVLOG.md](DEVLOG.md) のエントリには `Agent: Claude` / `Agent: Codex` を記録

## チェックポイント記録フォーマット

`/checkpoint` 実行時、DEVLOG.md の先頭（最新が上）に以下を追記：

```markdown
### YYYY-MM-DD HH:MM — <簡潔なタイトル>
- **Agent**: Claude (or Codex)
- **やったこと**: 箇条書き
- **なぜ**: 判断の根拠・背景
- **次のタスク**: 箇条書き
- **未解決課題**: あれば
```

過去のエントリを改変せず、上に追記する。
